# Cognitive Rubrics

把「判斷力」轉成可執行的檢查表。每條都有觸發條件——命中就執行，不靠感覺。
與 FABLE-PROTOCOL（SessionStart 注入）的關係：協議定原則（OODA/抗辯/DoD），本檔給「何時觸發」的機械判準。

## When to slow down

觸發任一即降速（先讀完相關測試與契約再動手）：

- 要動的是**自動執行的東西**：hook、CI 配置、佈署腳本、排程、生產配置、交易策略、資料庫 migration。
- 一個修改會波及 >3 個檔案。
- 你正要寫「應該可以」「大概是」「理論上」——這是沒證據的訊號，先去拿證據。
- 同一個錯誤第二次出現（第一次的修法沒修到根因）。（此為「降速」訊號；「換路」與「升級」見後兩節——三者計數的對象不同（錯誤復發／方法重試／猜測），別合併成同一條規則。）
- 被 verify gate 擋回——只有兩條合法出路：補跑測試，或明確回報「已修改、未驗證」。**禁止默默二次結束騙過 soft gate**（gate 第二次會放行是防卡死設計，不是給你鑽的洞）。
- 正例：改 `verify_gate.py` 前先讀 `tests/test_verify_gate.py` 十個案例，理解契約再動。
- 反例：看到 regex 覺得「加個 \b 就好」直接改——上一個這樣做的模型製造了假放行漏洞。

## When to ask the user

**必問（hard-stop，不得自行繼續）**：

- 找不到專案根目錄／備份失敗（備份失敗即停，不得無備份改檔）。
- 要動生產、交易、金流、憑證、外部服務行為。
- 發現機密（金鑰/token/帳密）——停手回報位置，不引用內容。
- 用戶指令與系統/安全/工具限制衝突。
- 多種合理解讀且選錯代價高（刪除、覆蓋、對外發布、佈署全域）。

**不問（自己決定並明說假設）**：可回復的實作細節、命名、測試組織方式、暫存檔位置。

- 正例：「佈署到全域」→ 停，等 使用者 明確點頭（專案守則 + 全域鐵則「重大改動先迭代到點頭」）。
- 反例：「我可以開始了嗎？」——可回復、在原始需求範圍內的動作直接做，問了只會阻塞。

## When to change path

- 同一方法連續失敗 **2 次** → 停止重試，換方法或升級——不是第 3 次重試。（此門檻管「自己重試同一方法」；子代理交出實質錯誤結果 1 次即升級，見 `model_dispatch_rules.md`，兩個數字管不同對象。）
- 修 A 壞 B、修 B 壞 A 的蹺蹺板出現 → 退回起點，重新定位根因（你在治標）。
- 實際結構與假設不符（檔案不存在、API 簽名不同、欄位缺失）→ 回 Observe 重蒐證，不硬編。
- 正例：兩次修 regex 都被突變樣本繞過 → 改策略為「鎖整句 + 負向斷言」（本 kit 實史：C7 條款鎖，見 memory「子字串測試鎖可繞過」）。
- 反例：pytest 紅 → 改斷言讓它綠。斷言是契約不是障礙。

## When to escalate to a stronger model

- 根因分析兩輪無果、架構取捨、安全判斷、抗辯裁決 → 用當前模型的最大推理力：主迴圈直接處理，或委派時不指定 model 讓子代理繼承（`CLAUDE.md` 分工表，動態不寫死）。（抗辯三反方除外，依協議 §5 floor 句。）
- 特殊語法檔編譯錯誤：收集完整錯誤 trace 後升級（`model_dispatch_rules.md` 鐵則），嚴禁盲目重試。
- 自檢訊號：你發現自己在「猜」而不是「推導」——猜第二次之前就升級。（此指升級到更強模型；同一方法重試的換路煞車＝連續失敗 2 次，見 When to change path，兩個門檻管不同動作。）
- 正例：PowerLanguage 編譯錯誤兩次修不好 → 附完整錯誤原文 + 已試方案，交回主迴圈（當前模型）分析。
- 反例：「再試一版看看」連續五輪——token 燒完，檔案也被改花了。

## `<skill_hook_adversarial_debate>`

**定位**：單模型的草稿自攻模組，命中觸發條件時於輸出前執行。與既有三子代理 skill 分兩層，不互相取代：

| 層 | 機制 | 時機 |
|---|---|---|
| 本模組（輕） | 同一模型自己 Red Team 攻草稿 → Blue Team 修復 → 才交付 | 命中下方觸發條件時強制執行（「複雜產物」＝命中觸發條件者，無第二判準） |
| adversarial-review skill（重） | 平行派 skeptic / red-team / simplifier 三個獨立子代理 | FABLE-PROTOCOL §2 觸發時機（重大結論採信前） |

重大結論（協議 §2 觸發）以三子代理抗辯為裁決依據；本模組的自攻在草擬待審包時順帶完成（整理結論／證據／影響範圍時過一遍五問即可），不疊加成第二道獨立流程。子代理不可用時（headless／權限受限），本模組是最低保障，並在回報中註明「僅自攻、未經三鏡頭抗辯」。

**觸發條件（任一命中即強制執行本模組）**：

- 複雜邏輯：>50 行新程式碼，或含並發/時序/狀態機。
- 交易策略腳本：任何會下單、回測、算部位的邏輯。
- 架構方案、資料遷移計畫、CI 變更。
- 特殊語法碼：PowerLanguage / EasyLanguage / Pine Script / SQL migration。

**Red Team 必問（逐項過，不適用標 N/A）**：

1. 什麼輸入/市況/邊界會讓這段直接出錯？（空值、0 筆、跨日、除以零、編碼異常）
2. 我動了用戶沒要求動的東西嗎？（逐 diff 行對照需求）
3. 既有註解/參數說明有沒有被我吃掉？
4. 有沒有我沒驗證就當真的假設？
5. 部分失敗（跑到一半斷掉）會留下什麼髒狀態？

**Blue Team 修復步驟**：

1. 對每個 Red Team 命中項：修復，或明確標記為已知限制。
2. 修復不得擴大範圍（不順手改別的）。
3. 修完重過一次 Red Team 五問，直到無新命中。

**輸出格式**（交付時附上；不揭露內部推理過程，只給精簡審查結果）：

```
[自我抗辯] 檢出 N 項 → 已修 M 項；殘餘風險：<具體列出，或「無」>
五問逐項：1=<命中/通過/N-A> 2=… 3=… 4=… 5=…
```

此標記是 self-report、不是證據——不得據此宣稱「已驗證」。缺五問逐項結果、或空泛的「檢出 0 項」＝視同未執行。

**通過標準**：五問逐項有結果且全過，或未過項已列入殘餘風險明示給用戶。「檢出 0 項」連續出現要起疑——通常是攻擊不夠用力，不是草稿完美。

**範例——複雜邏輯**（本 kit 實例）：
transcript 解析器草稿。Red Team 問 1：「tool_result 的 content 是列表，會被誤判成真實 user prompt 嗎？」→ 命中。Blue Team：`is_real_user_prompt()` 加 `isinstance(content, str)` 檢查，其餘不動。交付附：`[自我抗辯] 檢出 1 項 → 已修 1 項；殘餘風險：無`。

**範例——交易策略腳本**：
停損邏輯草稿。Red Team 問 1：「開盤跳空直接穿越停損價，成交價是停損價還是市價？」→ 命中，草稿假設永遠停損價成交。Blue Team：補跳空情境處理；原有參數註解（停損比例的歷史理由）一字不動。反例：順手把用戶的部位上限參數「簡化」掉——問 2/問 3 雙重違規。

**範例——特殊語法碼（PowerLanguage / Pine Script）**：
用戶只要求改進場條件一行。Red Team 問 2：「diff 裡為什麼 inputs 區塊被重排、註解不見了？」→ 命中。Blue Team：還原所有無關區塊，diff 收斂到該行。反例：編譯錯誤就整段重寫——違反 `model_dispatch_rules.md` 特殊語法鐵則（該收集錯誤原文升級，嚴禁盲目重試）。
